
## 언어 모델 자체의 한계
언어 모델 자체는 텍스트를 처리하고 텍스트를 반환할 뿐, 파일을 읽거나 명령어를 실행할 수는 없습니다. 독립형 언어 모델에 파일을 읽으라고 요청하면, 해당 기능이 없다고 응답할 것입니다.

## 코딩도우미 와 언어모델의 콜라보레이션
1. 사용자 : "main.go 파일에는 어떤 코드가 있나요?" -> 코딩 도우미
2. 코딩 도우미 : 사용자의 요청 + 도구 사용 지침을 추가 -> 언어 모델
3. 언어 모델 : "ReadFile: main.go" 라고 응답 -> 코딩 도우미
4. 코딩 도우미 : main.go 에는 이런 이런 내용이 있습니다. 라고 응답 -> 언어 모델
5. 언어 모델 : 파일 내용을 기반으로 최종 답변을 제공합니다. -> 사용자

## 언어 모델은 다양한 도구를 필요로 함

## 클로드 코드의 강점
Claude Code는 파일 읽기, 코드 작성, 명령 실행, 디렉터리 관리와 같은 
일반적인 개발 작업을 처리하는 포괄적인 내장 도구 세트를 제공합니다. 
하지만 Claude Code를 진정으로 강력하게 만드는 것은 
이러한 **도구들을 지능적으로 조합**하여 복잡하고 여러 단계를 거치는 문제를 해결하는 능력입니다.

## CLAUDE.MD 파일
클로드와 함께 코딩 프로젝트를 진행할 때는 컨텍스트 관리가 매우 중요합니다. 프로젝트에는 수십 개 또는 수백 개의 파일이 있을 수 있지만, 클로드는 효과적인 작업을 위해 필요한 정보만 있으면 됩니다. 불필요한 컨텍스트가 너무 많으면 오히려 클로드의 성능이 저하되므로, 클로드가 관련 파일과 문서를 참조하도록 안내하는 방법을 배우는 것이 필수적입니다.

클로드에게 코드베이스를 안내하며 중요한 명령어, 아키텍처 및 코딩 스타일을 알려줍니다.
클로드에게 구체적이거나 사용자 지정적인 지시를 내릴 수 있습니다.

CLAUDE.md - /init 명령으로 생성, 소스 코드 관리 시스템에 커밋, 다른 엔지니어들과 공유됨
CLAUDE.local.md - 다른 엔지니어와 공유되지 않으며, Claude에 대한 개인적인 지침 및 사용자 지정 설정이 포함되어 있습니다.
~/.claude/CLAUDE.md - 컴퓨터의 모든 프로젝트에서 사용되며, Claude가 모든 프로젝트에서 따라야 할 지침이 포함되어 있습니다.

## "@"+파일경로 기능
@파일경로 사용 시
  - 파일 내용이 즉시 자동으로 대화 컨텍스트에 포함됨
  - Claude가 별도 Read 툴 호출 없이 바로 내용을 볼 수 있음
  - 응답이 빠르고, 툴 승인 과정도 없음

그냥 파일 경로만 언급 시
  - 파일이 자동으로 읽히지 않음
  - Claude가 Read 툴을 직접 호출해서 파일을 읽어야 함
  - 툴 호출 승인 단계가 추가됨

실용적인 차이
  # @를 쓸 때 → 파일 즉시 로드, 바로 분석 가능
  "@src/lib/auth.ts 이 파일 설명해줘"
  
  # @없이 쓸 때 → Claude가 Read 툴로 파일을 따로 읽어야 함 
  "src/lib/auth.ts 이 파일 설명해줘"

결론 : 특정 파일을 기준으로 질문하거나 수정 요청할 때는 @를 붙이는 게 더 효율적입니다.
